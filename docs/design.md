# AI 视觉对话助手设计文档

## 1. 项目概述

本项目实现一个运行在浏览器中的 AI 视觉对话助手。用户开启摄像头与麦克风后，可以通过语音或文本向 AI 提问；当问题需要视觉信息时，系统截取摄像头当前画面，调用视觉模型生成画面描述，再将视觉描述、用户问题和对话历史交给聊天模型生成回答。

AI 回复会显示在聊天区域，并通过浏览器语音合成播放。页面中的 AI 角色会根据交互阶段切换为待机、倾听、思考和说话状态。

项目重点关注以下三个方面：

1. **视觉理解准确性**：确保视觉模型获得与用户问题相关、尺寸合理且格式有效的当前画面。
2. **语音交互自然度与流畅性**：让语音输入、AI 回复、语音播放和角色状态形成连续反馈。
3. **端云协同成本控制**：将摄像头预览、视觉判断、截图压缩和语音播放放在浏览器本地完成，只在必要时调用云端模型。

---

## 2. 总体设计

### 2.1 架构设计

项目采用前后端分离架构：

```text
摄像头 / 麦克风
        │
        ▼
React 前端
├── 摄像头实时预览
├── 浏览器语音识别
├── 视觉意图判断
├── 当前帧截图与压缩
├── AI 回复语音播放
└── 角色状态动画
        │
        ├── POST /api/vision
        └── POST /api/chat
        │
        ▼
FastAPI 后端
├── 视觉描述生成服务
├── 视觉缓存与调用统计
├── 对话模型服务
├── System Prompt 注入
└── 短期会话记忆
        │
        ▼
OpenAI 兼容聊天模型 / 视觉模型
```

### 2.2 核心交互流程

1. 用户开启摄像头，视频流只在浏览器本地实时预览。
2. 用户点击语音输入并说出问题，浏览器将语音识别为文本。
3. 文本进入统一发送流程，角色由 `listening` 切换为 `thinking`。
4. 前端判断问题是否需要视觉信息。
5. 普通问题直接调用聊天接口。
6. 视觉问题截取当前摄像头画面，将图片缩放、压缩后调用视觉接口。
7. 后端优先检查视觉描述缓存；没有可用缓存时才调用视觉模型。
8. 后端将视觉描述、用户问题和对话历史组合后调用聊天模型。
9. 前端展示 AI 回复并自动播放语音，角色切换为 `talking`。
10. 语音播放结束后，角色回到 `idle`。

## 3. 视觉理解设计

### 3.1 设计目标

视觉模型需要回答用户当前关心的问题，例如：

- “你看看桌上有什么？”
- “看一下我手里拿的是什么。”
- “这个东西是什么颜色？”

系统不追求连续视频分析，而是关注用户提问瞬间的当前画面。这样可以降低延迟、模型调用次数和图片传输成本。

### 3.2 实现方式

- 摄像头通过浏览器 `getUserMedia` 获取。
- 视频流仅在浏览器本地预览，不连续上传。
- 前端使用关键词规则判断视觉意图。
- 视觉问题触发时，通过 Canvas 截取当前帧。
- 图片统一转为 JPEG，最大宽高不超过 `768px`。
- 图片通过逐级降低质量压缩到不超过 `500KB`。
- 后端校验图片大小、MIME 类型和实际文件签名。
- 视觉模型只生成简短画面描述，不直接负责完整聊天回答。
- 聊天模型基于视觉描述和用户问题生成最终回复。

### 3.3 准确性考虑

实际采用了以下措施提升视觉回答可靠性：

1. **使用当前帧而不是旧图片**：视觉问题发送时立即截图。
2. **保留足够分辨率**：最大尺寸设置为 `768px`，在成本和识别能力之间取得平衡。
3. **低温度视觉提示词**：要求模型描述物体、颜色、位置和明显状态，不猜测无法确认的信息。
4. **视觉描述与用户问题一起注入聊天上下文**：防止聊天模型忽略画面信息。
5. **图片格式校验**：避免无效图片进入视觉模型。

### 3.4 当前限制

- 视觉意图判断基于关键词规则，无法覆盖所有自然语言表达。
- 缓存按照时间复用最近一次视觉描述，没有判断画面是否已经发生明显变化。
- 当前采用单帧理解，不适合动作识别和持续事件分析。
- 真实识别准确性取决于使用的视觉模型、光照、摄像头质量和画面遮挡。

## 4. 语音交互设计

### 4.1 语音输入

项目使用浏览器提供的语音识别能力，将语音转换为文本。识别出的文本与手动输入文本进入同一套消息发送流程，因此两种输入方式具有一致的视觉判断、聊天和错误处理行为。

语音识别期间，角色显示为 `listening`。识别完成并发送问题后，角色进入 `thinking`。

### 4.2 AI 回复语音播放

项目使用浏览器 `speechSynthesis` 播放 AI 回复：

- AI 回复生成后自动播放。
- 播放开始时角色进入 `talking`。
- 播放结束后角色回到 `idle`。
- 收到新回复时停止上一段语音，避免声音重叠。
- 用户可以关闭“回复语音”开关。
- 浏览器不支持语音播放时，仍然显示文本回复。

### 4.3 自然度与流畅性考虑

实际采用了以下设计：

1. **统一语音和文本主流程**：避免两种输入方式产生不同交互行为。
2. **明确状态反馈**：页面显示“正在理解当前画面”“正在生成回答”“正在播放 AI 回复”等状态。
3. **角色动画联动**：通过 `idle`、`listening`、`thinking`、`talking` 提供即时反馈。
4. **防止重复请求**：处理上一条消息时暂时禁用语音输入和发送按钮。
5. **错误后可继续操作**：语音识别或接口失败不会导致页面白屏，用户可以改用文本继续提问。

### 4.4 当前限制

- 浏览器语音识别和语音合成的可用性取决于操作系统和浏览器。
- 当前无法直接使用自定义角色音色。
- 当前是“语音识别完成后发送”，不是实时流式语音对话。
- AI 回复不是流式生成，需等待完整文本返回后才能开始播放。

## 6. 运营成本控制设计

### 6.1 思考过的成本控制技巧

设计阶段考虑过以下方案：

1. 连续上传视频流，由多模态模型持续理解画面。
2. 固定时间间隔截图，例如每 3 秒分析一次。
3. 用户发送任何问题时都上传当前截图。
4. 根据用户问题判断是否需要视觉信息，只在必要时截图。
5. 在浏览器本地进行截图缩放和图片压缩。
6. 缓存最近一次视觉描述，短时间内重复问题复用缓存。
7. 使用图片相似度判断画面变化后再决定是否调用模型。
8. 让视觉模型只生成简短描述，再交给文本模型回答。
9. 使用同一个高能力多模态模型完成所有聊天和视觉任务。
10. 文本聊天和视觉理解使用不同模型，根据任务选择更便宜的模型。
11. 限制视觉模型和聊天模型的输出 Token。
12. 开发和测试阶段提供 Mock 模式。
13. 使用浏览器本地语音识别和语音合成，避免额外语音 API 费用。
14. 对模型请求增加超时、并发限制和调用统计。

### 6.2 实际采用的成本控制策略

| 策略 | 实际实现 | 成本收益 |
| --- | --- | --- |
| 事件触发式视觉理解 | 只有开启自动视觉理解且检测到视觉问题时才截图。 | 避免普通聊天和页面停留产生视觉调用。 |
| 不上传连续视频 | 摄像头视频流仅在浏览器本地预览。 | 将持续云端视频分析成本降为零。 |
| 本地图片缩放 | 上传前将图片最大宽高限制为 `768px`。 | 减少网络传输和视觉输入 Token。 |
| 本地图片压缩 | JPEG 质量逐级降低，确保图片不超过 `500KB`。 | 减少请求大小和模型处理成本。 |
| 视觉描述缓存 | 最近描述默认缓存 `5` 分钟。 | 短时间重复视觉问题不再次调用模型。 |
| 简短视觉输出 | 视觉模型默认最多输出 `120` Token。 | 控制视觉模型输出费用和延迟。 |
| 视觉与聊天解耦 | 视觉模型生成描述，文本模型负责最终回答。 | 可以为两类任务分别选择性价比模型。 |
| 浏览器本地语音能力 | 使用浏览器语音识别和 `speechSynthesis`。 | 当前版本无需支付 STT 和 TTS API 费用。 |
| Mock 模式 | 无 API Key 时返回本地固定结果。 | 开发、调试和自动化测试不产生模型费用。 |
| 调用次数可观察 | 页面和日志记录视觉模型调用次数与缓存复用。 | 方便验证成本策略是否真正生效。 |
| 请求期间禁用重复输入 | 消息处理期间禁用发送和语音输入。 | 减少重复点击导致的重复请求。 |

### 6.3 未采用方案及原因

| 方案 | 未采用原因 |
| --- | --- |
| 连续视频流上传 | 成本和带宽占用过高，且多数对话不需要持续视觉信息。 |
| 定时截图分析 | 用户不提问时也会产生无意义模型调用。 |
| 每条消息都附带图片 | 普通聊天会浪费视觉模型费用，并增加响应延迟。 |
| 图片相似度缓存 | 第一版优先采用简单的时间缓存；相似度计算和缓存键设计留作后续优化。 |
| 云端语音识别与 TTS | 浏览器能力已满足原型需求，云端方案会增加费用和接入复杂度。 |
| 单一大模型完成全部任务 | 不利于按任务选择低成本模型，也不利于独立统计视觉调用。 |

## 7. 端云协同设计

项目将适合在本地完成、且高频发生的工作放在浏览器端，将需要模型能力的工作放在云端。

### 7.1 浏览器端负责

- 摄像头和麦克风权限管理。
- 摄像头实时预览。
- 用户语音识别。
- 视觉意图判断。
- 当前帧截取、缩放和压缩。
- AI 回复语音播放。
- 角色动画和交互状态提示。

### 7.2 后端与云端负责

- 图片安全校验。
- 视觉描述缓存和调用统计。
- 视觉模型调用。
- System Prompt 和视觉上下文注入。
- 对话模型调用。
- 短期对话记忆。

### 7.3 设计收益

- 高频、低复杂度操作在本地执行，减少云端调用。
- 原始视频流不离开浏览器，只上传必要的当前帧。
- 模型服务与前端交互解耦，便于替换供应商和调整成本。
- 语音、视觉和文本能力可以独立降级，单个能力失败不会导致整个页面不可用。

## 9. 总结

本项目最终实现了一个完整的视觉语音对话原型：用户可以通过语音或文本提问，AI 能够在需要时查看摄像头当前画面、结合对话历史生成回复，并通过语音和角色动画进行反馈。

在成本控制方面，项目没有采用连续视频理解，而是通过视觉意图判断、事件触发截图、本地图片压缩、短期视觉缓存、视觉与聊天模型解耦、浏览器本地语音能力和 Mock 模式降低运营成本。

该方案牺牲了连续视频动作理解和高级语音能力，但在视觉准确性、交互流畅性、实现复杂度与运营成本之间取得了适合当前原型和评审演示的平衡。
